# Structured Generation

Pie can mask the model's logits each step so the *only* tokens it can sample
are ones that keep the output valid against a formal grammar. The
declarative entry point is `Schema`; the lower-level escape hatch is
`GrammarConstraint`; for fully custom logic you implement the `Constrain`
trait yourself. Multiple constraints AND together.

## The easy path: `Schema`

```rust
use inferlet::{Schema, inference::Sampler};

// Force any valid JSON
let json: serde_json::Value = ctx
    .generate(Sampler::greedy())
    .with_schema(Schema::Json)?
    .with_max_tokens(256)
    .collect_json()
    .await?;

// JSON Schema
let schema = r#"{"type":"object","properties":{"city":{"type":"string"}},"required":["city"]}"#;
let text = ctx
    .generate(Sampler::greedy())
    .with_schema(Schema::JsonSchema(schema))?
    .with_max_tokens(128)
    .collect_text()
    .await?;

// Regex
ctx.generate(Sampler::greedy())
   .with_schema(Schema::Regex(r"\d{4}-\d{2}-\d{2}"))?;

// EBNF (Lark-style)
ctx.generate(Sampler::greedy())
   .with_schema(Schema::Ebnf(my_grammar))?;

// Pre-compiled grammar (compile once, reuse)
let grammar = inferlet::inference::Grammar::from_ebnf(my_grammar)?;
ctx.generate(Sampler::greedy())
   .with_schema(Schema::Grammar(&grammar))?;
```

`Schema` variants:

| Variant | Constrains output to |
|---|---|
| `Schema::Json` | Any valid JSON. |
| `Schema::JsonSchema(s)` | JSON satisfying the JSON-Schema string `s`. |
| `Schema::Regex(p)` | Strings matching pattern `p`. |
| `Schema::Ebnf(g)` | A custom EBNF grammar (Lark format). |
| `Schema::Grammar(&g)` | A pre-compiled `Grammar` (build it once and reuse). |

`with_schema` returns `Result` because schema parsing can fail; everything
after `?` is masked-sampling-fast.

:::tip
Pair grammars with `Sampler::greedy()` (or low-temperature top-p). Once the
mask narrows the distribution to a handful of valid tokens, stochastic
sampling rarely improves quality and can hurt throughput by lowering
acceptance under speculative decoding.
:::

## `collect_json::<T>()`

The macro-friendly path: derive `Deserialize` and `JsonSchema` on your
output type and let the SDK infer the schema.

```rust
use schemars::JsonSchema;
use serde::Deserialize;

#[derive(Deserialize, JsonSchema)]
struct Plan { steps: Vec<String> }

let plan: Plan = ctx
    .generate(Sampler::greedy())
    .with_max_tokens(256)
    .collect_json::<Plan>()
    .await?;
```

`collect_json` builds the JSON schema for `T`, applies it as a constraint,
samples, and parses — all in one call. (It errors if you also call
`with_schema` because it owns the schema.)

## Lower level: `GrammarConstraint`

For when you want to build a constraint and reuse it across multiple
streams (e.g., the same EBNF every turn of an agent loop), construct a
`GrammarConstraint` and feed it to `with_constraint`:

```rust
use inferlet::context::GrammarConstraint;

let gc = GrammarConstraint::from_ebnf(my_grammar, &model)?;

let out = ctx
    .generate(Sampler::greedy())
    .with_constraint(gc)
    .with_max_tokens(256)
    .collect_text()
    .await?;
```

Constructors mirror `Schema`:

```rust
GrammarConstraint::json(&model);
GrammarConstraint::from_grammar(&grammar, &model);
GrammarConstraint::from_json_schema(s, &model)?;
GrammarConstraint::from_regex(p, &model)?;
GrammarConstraint::from_ebnf(g, &model)?;
```

## Custom constraints: `Constrain`

For non-grammar logic — banned-tokens lists, learned masks, dynamic
constraints — implement the trait:

```rust
pub trait Constrain: Send {
    /// Advance internal state with the tokens just accepted, then return the
    /// BRLE-encoded logit mask for the next position.
    /// Returning &[] means "no restriction" (transparent during composition).
    fn step(&mut self, accepted: &[u32]) -> &[u32];
}
```

`accepted` is `&[]` on the first step; thereafter it's whatever the sampler
(or the verifier in spec-decode) committed since the last `step` call. The
mask uses BRLE: `[run_of_false, run_of_true, run_of_false, …]`.

Multiple constraints AND together — every constraint applied via
`with_schema` or `with_constraint` contributes a mask and they're combined
before each forward pass.

## How it fits in a step

1. The stream calls `step(accepted)` on every active constraint.
2. The masks AND together (BRLE intersection).
3. The combined mask is attached to the `ForwardPass` so the sampler only
   sees valid tokens.
4. Whichever tokens the sampler returns are fed back as `accepted` to the
   constraints on the next step.

This runs entirely inside the forward-pass loop — no rejection, no retry.
