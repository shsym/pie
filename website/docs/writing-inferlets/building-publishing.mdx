# Building & Publishing

Source → WASM → Pie registry. The toolchain is `bakery` (a Python CLI in
`sdk/tools/bakery`); for raw `cargo` users it's a thin wrapper.

## Anatomy of an inferlet project

```
my-inferlet/
├── Cargo.toml          # Rust crate manifest
├── Pie.toml            # inferlet manifest (read at compile time + by `pie run`)
└── src/
    └── lib.rs          # contains #[inferlet::main]
```

`Cargo.toml`:

```toml
[package]
name = "my-inferlet"
version = "0.1.0"
edition = "2024"

[lib]
crate-type = ["cdylib"]      # required: produces a WASM component

[dependencies]
inferlet = "0.2"
serde = { version = "1.0", features = ["derive"] }

[profile.release]
lto = true
```

`Pie.toml`:

```toml
[package]
name = "my-inferlet"
version = "0.1.0"
description = "A helpful chat assistant"
authors = ["You <you@example.com>"]

[runtime]
core = "^0.2.0"
mcp  = "^0.2.0"            # only if you use inferlet::mcp::*

[parameters]
prompt     = { type = "string", description = "User prompt" }
max_tokens = { type = "int", optional = true }
```

The `inferlet::main` macro reads `Pie.toml` for the package name during
the build (`CARGO_MANIFEST_DIR/Pie.toml`).

## Building

### With cargo

```bash
rustup target add wasm32-wasip2     # one-time
cargo build --target wasm32-wasip2 --release
# output: target/wasm32-wasip2/release/my_inferlet.wasm
```

### With bakery

```bash
pip install -e sdk/tools/bakery     # one-time, from a Pie checkout

# Rust (auto-detects Cargo.toml)
bakery build ./my-inferlet -o my-inferlet.wasm

# TypeScript / JavaScript
bakery build ./my-ts-inferlet -o my-inferlet.wasm
bakery build ./index.ts -o single-file.wasm
```

| Flag | Description |
|---|---|
| `-o, --output <FILE>` | Output `.wasm` path. Required. |
| `--debug` | Debug build (JS/TS only — includes source maps). |

When developing inside a Pie checkout, set `PIE_SDK=/path/to/pie/sdk` so
bakery can find the SDK libraries.

### Scaffolding from scratch

```bash
bakery create my-inferlet            # Rust template (default)
bakery create my-inferlet --ts       # TypeScript template
```

The templates lay down `Cargo.toml` (or `package.json`), `Pie.toml`, and
a working `lib.rs` / `index.ts` against the current SDK.

## Running locally

```bash
# Local build — both files are required
pie run \
  --path  target/wasm32-wasip2/release/my_inferlet.wasm \
  --manifest ./Pie.toml \
  --prompt "Hello"

# After publishing, run from the registry
pie run my-inferlet --prompt "Hello"
pie run my-inferlet@0.1.0 --prompt "Hello"     # pinned
```

CLI flags after the manifest become the JSON `Input` (with int/float/bool
inference). See [`pie run`](../cli#run) for the full grammar.

## Publishing

```bash
# 1) Authenticate (GitHub OAuth)
bakery login

# 2) Publish — uploads the wasm + manifest to the registry
bakery inferlet publish ./my-inferlet
```

| Subcommand | Description |
|---|---|
| `bakery inferlet search <query>` | Search published inferlets. |
| `bakery inferlet info <name>` | Show metadata for a published inferlet. |
| `bakery inferlet publish <path>` | Build and publish. |

Once published, the inferlet is launchable from any Pie engine connected
to the same registry:

```bash
pie run my-inferlet --prompt "Hello"
```

```python
proc = await client.launch_process("my-inferlet@0.1.0",
                                    input={"prompt": "Hello"})
```

## Version resolution

If a name has no `@version`, the client (Python / JS / Rust) calls the
registry's `resolve_version` to pick the latest. Pin a version when you
need reproducibility:

```python
await client.launch_process("my-inferlet@0.1.0", input=...)
```

Or pre-resolve and reuse the result.
