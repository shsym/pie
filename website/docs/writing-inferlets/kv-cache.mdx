# KV Cache Control

Pie's KV cache is page-based and copy-on-write. The high-level `Context`
hides this most of the time, but you can drop down to the WIT resource
when you need fork / save / page-level control.

## Page model

Each context's KV state is a chain of fixed-size *pages* (`page_size`
tokens per page). Pages are either *committed* (immutable, shareable
across forks) or *working* (mutable, in-flight).

```
┌─────────┐ ┌─────────┐ ┌─────────┐ ┌────────────┐
│ Page 0  │ │ Page 1  │ │ Page 2  │ │  Page 3    │ ← working
│ (full)  │ │ (full)  │ │ (full)  │ │ (partial)  │
└─────────┘ └─────────┘ └─────────┘ └────────────┘
 committed   committed   committed     working
```

Inspecting state via the SDK wrapper:

```rust
ctx.page_size();         // u32 — tokens per page
ctx.seq_len();           // u32 — total tokens (committed + working)
ctx.buffer();            // &[u32] — SDK-side tokens not yet flushed to a forward pass
```

For lower-level counts (committed / working page counts, working-page
token count), reach into the WIT resource:

```rust
let raw = ctx.inner();                       // &RawContext
let committed = raw.committed_page_count();  // u32
let working   = raw.working_page_count();    // u32
let working_tokens = raw.working_page_token_count();
```

## Page operations

The SDK calls these for you during `flush()` / `generate()`. You only need
them for advanced patterns (custom forward-pass loops, sliding windows).

```rust
let raw = ctx.inner();

raw.reserve_working_pages(n)?;   // pre-allocate n working pages
raw.commit_working_pages(k)?;    // promote k full working pages to committed
raw.release_working_pages(n);    // free n working pages
raw.truncate_working_page_tokens(n); // drop the last n tokens (rollback)
```

`commit_working_pages(k)` makes the first `k` working pages immutable;
forks of this context can now share them.

## Forking

`ctx.fork()` is a copy-on-write clone. Committed pages are shared; working
pages are duplicated:

```rust
let mut child = ctx.fork()?;
child.user("…");
child.cue();
```

```
 fork():

 Original ── Page 0 ── Page 1 ── Page 2 ── Page 3 (working)
                │         │         │
 Child    ─────┘─────────┘─────────┘──── Page 3' (copied)

 ● Pages 0–2 are committed, shared, read-only.
 ● Page 3' is a fresh copy — divergent writes go here.
 ● O(1) fork; only divergent pages cost memory.
```

Use it for:

- **Branching generation** — try multiple continuations from one prefix.
- **Tree of thought / best-of-N** — explore in parallel.
- **Tool-call retries** — fork before the call, fall back if it fails.

## Prefix caching

The classic pattern: build a prompt once, save it, and fork from the
saved snapshot for each request.

```rust
// One-time setup — compute & cache the system prompt.
let mut prefix = Context::new(&model)?;
prefix.system("You are a helpful assistant. <long preamble…>");
prefix.flush().await?;
prefix.save("system-prompt-v1")?;
```

Per-request:

```rust
let prefix = Context::open(&model, "system-prompt-v1")?;
let mut ctx = prefix.fork()?;

ctx.user("New user question");
ctx.cue();
let resp = ctx.generate(sampler).collect_text().await?;
```

`Context::open` clones the snapshot so multiple inferlets can share it.
`Context::take` *moves* the snapshot into the current context (no copy)
and removes the named slot — useful for one-shot resumption.

```
 ┌── "system-prompt-v1" (saved snapshot, immutable) ─┐
 │  Page 0  │  Page 1  │  Page 2  │      Page 3      │
 └─────────────────────────────────────────────────-─┘
       │            │              │
   fork() (req-1)                fork() (req-2)
```

## Sliding windows / eviction

For unbounded-length generation you need to drop early tokens periodically.
Three primitives compose:

- `raw.truncate_working_page_tokens(n)` — drops trailing tokens before
  they're committed (e.g., reject speculation tail).
- `raw.release_working_pages(n)` — frees newest working pages.
- The runtime's [page-trim optimization](./page-trim) — leading-page trim
  for sparse-attention masks (covered separately).

For an example of bounded-memory generation with eviction, see the
`windowed-attention` inferlet under `inferlets/windowed-attention/`.

## Persistence

| Method | What it does |
|---|---|
| `ctx.save(name)?` | Snapshot the chain under a user-chosen name. |
| `ctx.snapshot()?` | Snapshot under a runtime-generated name; returns the name. |
| `Context::open(model, name)?` | Clone an existing snapshot (snapshot stays). |
| `Context::take(model, name)?` | Move a snapshot into a fresh context (snapshot deleted). |
| `Context::delete(model, name)?` | Drop a saved snapshot. |

Saved contexts persist across inferlet runs as long as the engine is up.

## Market hooks

Each context places a *bid* per page-step. The SDK manages this for you,
but two operations are useful:

```rust
ctx.suspend()?;                 // give back all pages; resume later
ctx.bid(0.0);                   // 0.0 = use default truthful bid
let _g = ctx.yield_bid(Duration::from_secs(2)); // RAII low-bid window
```

Suspended contexts are restored automatically (highest-bid first) when
memory frees up.
