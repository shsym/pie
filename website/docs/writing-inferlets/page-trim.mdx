# Page-Trim

**Page-trim** is a runtime optimization that drops physical KV pages from
a forward pass when your attention mask renders them irrelevant. It runs
automatically — there's no API to call. This page explains when it
kicks in and how to write masks that get the most out of it.

## What it does

When you submit a request to the engine, every row of your attention mask
references a list of physical pages. Page-trim looks for *whole leading
pages* whose every row is fully masked out and:

1. Excludes those pages from the request's `kv_page_indices`.
2. Slices the corresponding bits out of every BRLE row before flattening.
3. Keeps any page that will receive new K/V writes this pass (those are
   protected from trimming).

The result is a compute graph that touches fewer KV pages — the sparse
mask actually saves work, instead of just zero-attending the masked
positions.

## Why it matters

Patterns like sliding-window attention or attention-sink eviction are
naturally expressed as "mask out the middle" — but a naïve attention
implementation still loads and attends through those pages. Page-trim
turns the mask into a compute saving.

It's especially useful for:

- **Long-context generation** with bounded memory: the cache *grows*
  but the per-step compute stays bounded by the window size.
- **Streaming summarization** that drops old context.
- **Custom pruning policies** implemented in the inferlet.

## When it kicks in

```
 Request masks (one per input token, BRLE):
 ┌────────────────────────────────────────────────────────┐
 │ row 0:  attend pages [0, 4, 5, 6]   (1, 2, 3 = masked) │
 │ row 1:  attend pages [0, 4, 5, 6]                      │
 │ row 2:  attend pages [0, 4, 5, 6, 7]                   │
 └────────────────────────────────────────────────────────┘

 Page-trim plan:
   pages 1, 2, 3 are fully masked across every row → trim them.
   pages 0, 4, 5, 6, 7 are kept.

 Wire-format payload after trim:
   kv_page_indices  = [0, 4, 5, 6, 7]
   per-row BRLE     = sliced to remove the bits covering pages 1–3
```

The fast path is "no trim possible" (causal masks, decode steps), and
that path allocates zero. So leaving page-trim enabled is free for
requests that don't benefit.

## Writing trim-friendly masks

Two ground rules make trim land:

- **Mask whole pages, not individual tokens.** A row that masks
  positions `5..10` inside a 16-token page won't trim that page;
  masking positions `0..16` will.
- **Apply the same mask shape to *every row* of the request.** Trim
  only fires when every row agrees that a page is dead. If row 0
  attends to page 1 but rows 1+ don't, page 1 stays.

The high-level inferlet idioms align with this naturally:

- **Sliding window with whole-page eviction** — release working pages
  for tokens before the window, mask the window in attention. The
  evicted pages disappear from the request entirely.
- **Attention sink** — keep a fixed prefix of pages live, mask the
  middle, keep the tail. Trim removes the middle.
- **Page-aligned sparse attention** — pre-aggregate context into
  page-sized chunks so masks align.

## Working with the SDK

You don't call anything special. `ctx.append(...)` / `ForwardPass`
already produce the request shape page-trim cares about. Two SDK
operations interact with the optimization:

| Operation | Effect |
|---|---|
| `raw.release_working_pages(n)` | Frees working pages up front; if you also mask them, trim drops them from the request. |
| `pass.attention_mask(&[brle, ...])` | The mask page-trim inspects. Same BRLE on every row → best chance to trim. |

The `inferlets/attention-sink/` and `inferlets/windowed-attention/`
examples show the pattern end to end.

For benchmarking, see `runtime/benches/page_trim_bench.rs`.

## Caveats

- **Pages receiving new K/V writes are never trimmed.** That includes
  the active working page on a generation step. So a "drop everything
  but the latest 16 tokens" mask still keeps the active page.
- **Trim is per-request.** Two batched requests with different masks
  don't share trim decisions; each is computed independently.
- Page-trim only affects *physical* pages on the wire. Your context's
  *logical* token positions don't change, and `position_ids` are
  preserved.
