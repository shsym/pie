# Generation

After filling a context with tokens, you generate new ones through a
`TokenStream` (raw tokens) or an `EventStream` (decoded chat events).

- **`TokenStream`** — yields `Vec<u32>` per `next()`, full control.
- **`EventStream`** — yields `Event::{Text, Thinking, ToolCall, Done, …}`,
  fused with the model's chat / reasoning / tool-use decoders.

## Basic generation

```rust
use inferlet::{Context, inference::Sampler};

let mut stream = ctx
    .generate(Sampler::top_p(0.6, 0.95))
    .with_max_tokens(256);

while let Some(tokens) = stream.next().await? {
    let text = ctx.model().tokenizer().decode(&tokens)?;
    print!("{text}");
}
```

`ctx.generate()` returns a [`TokenStream`](#tokenstream-builder). Each
`next()` runs one forward pass and returns the accepted tokens.

### Convenience collectors

```rust
// Vec<u32> of all generated tokens
let tokens = ctx.generate(sampler).with_max_tokens(256).collect_tokens().await?;

// Decoded text
let text = ctx.generate(sampler).with_max_tokens(256).collect_text().await?;

// Typed JSON output (also applies a Schema constraint internally)
#[derive(serde::Deserialize)]
struct Plan { steps: Vec<String> }
let plan: Plan = ctx.generate(sampler).collect_json().await?;
```

## Samplers

The `Sampler` enum chooses how a token is selected (or what the inferlet
*receives* — distributions, logits, etc.) at each position.

### Sampling modes

| Constructor | Returns | Notes |
|---|---|---|
| `Sampler::greedy()` / `Sampler::ARGMAX` | a token | Deterministic. Same as `top_p(0.0, 1.0)`. |
| `Sampler::top_p(temp, p)` | a token | Nucleus sampling: keep tokens up to cumulative `p`. |
| `Sampler::top_k(temp, k)` | a token | Sample from the top `k`. |
| `Sampler::min_p(temp, p)` | a token | Keep tokens with `prob ≥ p × max_prob`. |
| `Sampler::top_k_top_p(temp, k, p)` | a token | First top-`k`, then nucleus. |
| `Sampler::multinomial(temp, draws)` | a token | Plain multinomial after temperature scaling. |

### Output modes (no token selection)

These return data instead of choosing a token. Useful for best-of-N,
beam search, scoring, custom samplers.

| Constructor | Output | Notes |
|---|---|---|
| `Sampler::distribution(temp, k)` | top-`k` `(token_ids, probs)` | Keeps the temperature-scaled distribution. |
| `Sampler::raw_logits()` | packed `f32` bytes | Pre-softmax, untemperatured logits per position. Decode with `bytemuck::cast_slice::<u8, f32>(buf)`. |
| `Sampler::logprob(token_id)` | `Vec<f32>` (length 1) | `log p(token_id | context)`. |
| `Sampler::logprobs(ids)` | `Vec<f32>` (length K) | Multi-candidate logprobs in input order. |
| `Sampler::entropy()` | `f32` | Shannon entropy of the unscaled distribution. |

```rust
// Greedy
let s = Sampler::greedy();

// Nucleus
let s = Sampler::top_p(0.6, 0.95);

// Yes/no scoring
let yes = ctx.model().tokenizer().encode(" yes");
let no  = ctx.model().tokenizer().encode(" no");
let s = Sampler::logprobs(vec![yes[0], no[0]]);
```

`Sampler::default()` is `ARGMAX`.

### Reading the output

Generation returns an `Output` with a slot list — one entry per requested
position. For the typical "one sampler, one token" case there are
single-line accessors:

```rust
let mut stream = ctx.generate(Sampler::greedy()).with_max_tokens(256);
while let Some(_tokens) = stream.next().await? {
    // tokens already returned by `next()`
}

// Lower-level: examine a single forward pass output
let out = ctx.generate(Sampler::distribution(1.0, 32))
            .with_max_tokens(1)
            .collect_tokens().await?;
```

For non-token outputs, drive the stream manually and inspect the underlying
`Output` (see [Sampler outputs](./sampler-outputs) for the slot-list
details).

## EventStream

For chat applications, decode tokens into structured events:

```rust
use inferlet::Event;

let mut events = ctx
    .generate(Sampler::top_p(0.6, 0.95))
    .with_max_tokens(256)
    .decode()
    .with_reasoning()       // surface <think>…</think> as Event::Thinking
    .with_tool_use();       // surface tool calls as Event::ToolCall

while let Some(event) = events.next().await? {
    match event {
        Event::Text(s)              => print!("{s}"),
        Event::Thinking(s)          => print!("[think] {s}"),
        Event::ThinkingDone(_)      => println!("[/think]"),
        Event::ToolCall(name, args) => handle_tool(&name, &args),
        Event::Done(full)           => break,
        Event::Token                => {}, // raw token, no semantic meaning
    }
}
```

`EventStream` fuses a `TokenStream` with a `Decoder`. See
[Chat & Instruct](./chat-instruct) for decoder details and
[Tool calling](./tool-calling) for handling `ToolCall`.

## TokenStream builder

| Builder method | Description |
|---|---|
| `.with_max_tokens(n)` | Stop after `n` accepted tokens. |
| `.with_speculation(s)` | Plug in a custom [speculation](./speculative-decoding). |
| `.with_constraint(c)` | Apply a logit [mask](./structured-generation). |
| `.with_schema(s)?` | Constrain output to a JSON schema. |
| `.with_adapter(&adapter)` | Use a [LoRA adapter](./adapters) for this run. |
| `.with_zo_seed(seed)` | Zeroth-order optimizer seed (advanced). |
| `.with_horizon(n)` | Hint expected output length for bid planning. |

The terminal methods that consume the stream:

| Method | Returns |
|---|---|
| `.next().await?` | `Option<Vec<u32>>` — next batch of tokens, `None` when done. |
| `.collect_tokens().await?` | All tokens as `Vec<u32>`. |
| `.collect_text().await?` | Full decoded `String`. |
| `.collect_json::<T>().await?` | Deserialize the response into `T`. Auto-applies a derived schema. |
| `.decode()` | Convert into an `EventStream`. |

`stream.ctx()` exposes the underlying `&mut Context` so you can append
tool answers / extra fills mid-generation between `next()` calls.

## How a step works

Each `next()` runs one forward pass:

1. Tokens buffered in the context (from `system/user/cue/append/…`) are
   sent through the model with their position IDs.
2. If [speculation](./speculative-decoding) is on, draft tokens are added
   so the verifier can accept several at once.
3. The sampler runs on the output logits; active
   [constraints](./structured-generation) AND together to produce a mask
   first.
4. Accepted tokens are committed to the KV cache.
5. Stop tokens (model EOS, plus any extra you set) halt the stream.

For raw control over the forward pass, see
[Custom forward pass](./custom-forward-pass).
