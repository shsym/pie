# Kernel Dispatch

The `pie_kernels` package is the abstraction the **native driver** uses
to call platform-specific compute kernels. It hides FlashInfer (CUDA)
and pie's hand-written Metal kernels behind one import:

```python
import pie_kernels as ops

wrapper = ops.BatchAttentionWithAttentionSinkWrapper(...)
```

You don't normally interact with `pie_kernels` from an inferlet — it
lives below the SDK. This page explains the layout for anyone working
on the engine itself or debugging a kernel choice.

## Backend selection

`pie_kernels/__init__.py` picks the backend at import time:

```python
import torch
if torch.backends.mps.is_available():
    from . import metal as _backend         # Apple Silicon
else:
    import flashinfer as _backend           # CUDA
```

Symbol lookup falls through `__getattr__` to the selected backend, so the
public surface mirrors whichever backend is active. Metal-only and
FlashInfer-only kernels are isolated to their respective subpackages and
called from CUDA-guarded paths in the driver.

## Source layout

```
pie/src/pie_kernels/
├── __init__.py          # backend dispatch
├── attention.py         # batched attention dispatchers
├── sampling.py          # sampling kernels (multinomial / top-k / top-p / …)
├── fp4_quantization.py  # FP4 quantize / dequantize
├── fused_moe.py         # MoE expert routing kernels
├── rand_mv.py           # random matrix-vector ops
├── cuda/                # pie-owned CUDA / Triton kernels
│   ├── cuda_kernels.cu
│   ├── rand_mv*.py
│   └── ziggurat_tables.h
└── metal/               # pie-owned Metal kernels (MSL + driver)
    ├── _compiler.py
    ├── _wrappers.py
    ├── sampling.py
    ├── rand_mv.py
    └── kernels/
```

## Apple Silicon path

`pie_kernels.metal/` ships pie's own Metal kernels — sampling, attention
sink, RoPE, paged KV. The driver opens them through the dispatcher:

- `metal._compiler.py` JIT-compiles MSL on first use.
- `metal._wrappers.py` is the Python ↔ Metal binding.

## CUDA path

CUDA delegates most work to upstream
[FlashInfer](https://github.com/flashinfer-ai/flashinfer); the
`pie_kernels.cuda/` subpackage hosts the kernels FlashInfer doesn't
provide (custom RNG / matrix-vector for Zo, the ziggurat tables, etc.).

Pie pins `flashinfer-python` and friends through the `cu126` / `cu128`
extras — the version is locked in `pie/pyproject.toml`.

## When this matters

- **Choosing a GPU**: pie supports CUDA SM 7.5+ (Triton) on the SGLang
  driver path and FlashInfer-supported hardware on the native driver
  path. Apple Silicon runs only on the native driver.
- **Custom kernels**: the native driver loads
  `pie_kernels.{attention, sampling}` for its hot paths — extending or
  replacing those is how you'd ship a new attention variant or
  sampling scheme.
- **Quantization**: `fp4_quantization.py` powers the `weight_dtype =
  "float8" / "int4"` paths in `pie_driver.quantization`.
